## Context

“开口”V1 需要尽快首发微信小程序，但产品目标并不止于卡片联想训练。后续版本已经明确会扩展到 AI 辩论、演讲训练、语音分析、语音播报、难度选择和后台运营，因此当前设计不能围绕单一玩法直接堆表、堆接口。

当前已有多份产品与架构文档，已经确定的核心方向包括：
- 小程序为首发形态，使用微信账号体系完成绑定。
- V1 首发以卡片联想训练为主，4x4 卡片矩阵，任意两张卡均可形成训练。
- 后台、内容中心、语音能力和 AI 编排必须预留长期扩展能力。
- 当前工作区已经建立 OpenSpec 结构，但尚无已归档 specs，需要本 change 作为 V1 的第一套正式规范。

## Goals / Non-Goals

**Goals:**
- 用统一的训练会话模型承载 V1 卡片训练，并兼容后续多训练模式。
- 定义 V1 必需的用户、会员、训练、反馈、内容和后台边界。
- 确保词库来源支持预置内容，并预留 AI 动态生成与按领域生成的内容来源抽象。
- 将前台 API、后台 API 和数据模型统一到领域边界上。
- 让 V1 开发可以按任务分组逐步推进，而不是先做一堆孤立页面。

**Non-Goals:**
- 本 change 不实现 AI 辩论、演讲训练、完整 TTS 和语音评分能力。
- 本 change 不要求 V1 上线即具备完整自动词库审核系统。
- 本 change 不要求微服务化部署。
- 本 change 不要求一次做完整 RBAC 后台权限系统。

## Decisions

### 1. 采用模块化单体，而不是微服务
- 选择：V1 后端采用模块化单体，按用户、会员、内容、训练、AI、语音、分析、后台等领域拆分模块。
- 原因：当前业务变化快，V1 优先验证首发链路；微服务会显著增加开发和运维成本。
- 备选方案：直接拆微服务。
- 不选原因：当前团队和阶段不需要承担服务治理、链路追踪、分布式事务等复杂度。

### 2. 以 `training_session` 为平台主模型
- 选择：所有训练模式都围绕训练会话、训练提交、训练反馈展开。
- 原因：这样卡片训练、辩论训练、演讲训练都能复用统一主线，避免 `word1/word2` 绑死结构。
- 备选方案：先按卡片训练写专用表，后续再重构。
- 不选原因：后续一定返工，而且历史数据迁移成本高。

### 3. 统一内容中心，区分内容类型与来源类型
- 选择：建立 `content_item` 作为统一内容中心，并引入 `content_source_type` 区分 `preset / ai_generated / imported`。
- 原因：V1 虽以预设词库为主，但后续一定会接 AI 动态词库和按领域生成。
- 备选方案：V1 只做词语表，未来再加内容中心。
- 不选原因：内容模型会很快碎裂，后台管理和训练抽取都会变复杂。

### 4. AI 输出采用 schema 驱动，而不是单一固定字段
- 选择：训练反馈表同时保留通用字段和完整 `feedback_json`，并带 `feedback_schema`。
- 原因：卡片训练、辩论训练、演讲训练的反馈结构不同，但都需要统一归档和展示入口。
- 备选方案：只保留一套固定字段。
- 不选原因：未来每新增一个模式都要破坏现有结构。

### 5. 语音能力与内容生成能力都走独立适配层
- 选择：ASR、TTS、语音分析、内容生成都不直接写进页面或训练控制器，而是走 `speech` / `ai` 领域能力。
- 原因：未来供应商可能变，且同一能力会被多个训练模式复用。
- 备选方案：在卡片训练流程中直接调用某个服务商 SDK。
- 不选原因：耦合高，扩展差，更换供应商成本大。

### 6. 后台从 V1 就交付最小运营版本
- 选择：V1 最低交付后台登录、仪表盘、用户列表、订单列表、词语管理。
- 原因：没有后台，就无法做运营、内容管理和首发后的策略调整。
- 备选方案：V1 先不做后台，全靠查库。
- 不选原因：会显著拖慢迭代效率，也不利于内容运营。

## Risks / Trade-offs

- [风险] V1 为了长期扩展引入平台抽象，可能让首发实现看起来比“纯卡片小程序”更复杂。 → Mitigation：保持部署为模块化单体，只在模型和接口层做抽象，不做过重基础设施。
- [风险] AI 和 ASR 服务存在时延与稳定性波动。 → Mitigation：先跑通文本提交流程，ASR 和 AI 结果都提供失败重试与兜底文案。
- [风险] 预留动态词库后，可能诱发团队过早投入自动生成能力。 → Mitigation：V1 只保留来源类型和生成任务模型，不强行实现复杂审核流。
- [风险] 后台范围膨胀。 → Mitigation：把后台范围严格收敛为最小运营集，不做完整权限和复杂配置中心。

## Migration Plan

1. 先按平台级数据模型建最小必要表。
2. 优先实现微信登录、会员、卡片训练、AI 反馈主链路。
3. 后续补历史记录和最小后台。
4. 将内容中心默认跑在预置词库模式上，同时预留 `content_generation_task`，但不强制启用动态生成。
5. V1 上线后，如需扩展辩论、演讲或动态词库，直接基于当前模型追加模式和内容来源，不推翻主干。

回滚策略：
- 若 AI 或 ASR 服务不稳定，可临时回退到文本输入 + AI 文本分析模式。
- 若后台未完成，不阻断小程序首发，但内容维护和运营效率会下降。

## Open Questions

- V1 首发是否需要先支持后台发起内容生成任务但不对普通用户开放。
- 词语动态生成在上线前是否需要人工审核，还是允许先进入测试池。
- 小程序首发阶段是否只开放 `preset` 内容策略，`ai_generated` 仅保留后台和数据库能力。
