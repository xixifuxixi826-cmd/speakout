# 开口长期技术架构设计

## 1. 结论先说

当前 V1 的产品方案不需要推翻，但如果技术实现仍然按照“卡片训练专用系统”去设计，后续扩展到管理后台、词语难度、语音分析、语音播报、AI 辩论、演讲训练时，会很快遇到结构性问题。

正确方向不是推翻 V1，而是从现在开始把“开口”定义成一个**表达训练平台**，而不是单一的“卡片训练功能”。

结论分两层：

1. 产品形态不需要推翻
2. 技术架构需要从“单玩法实现”升级为“多训练模式可扩展架构”

## 2. 当前方案哪里不够扩展

如果按现在最直觉的方式去开发，常见风险有这些：

### 2.1 把训练记录写死成卡片训练结构
现在的记录里有 `word1`、`word2`、`round_id` 这些字段，适合卡片模式，但不适合：
- AI 辩论
- TED 跟读
- 演讲朗读
- 未来更多思维训练模式

一旦后续加入新训练模式，训练记录表会不断打补丁，最后出现：
- `word1`
- `word2`
- `debate_topic`
- `speech_script_id`
- `audio_score`
- `...`

这类表会非常难维护。

### 2.2 把 AI 反馈写死成一种输出结构
当前 AI 反馈结构主要适合“短表达点评”。

但未来新增模式后，反馈结构会明显不同：
- 辩论模式要看“立场明确、反驳力度、追问能力”
- 演讲模式要看“停顿、重音、感染力、朗读准确度”
- 跟读模式要看“语音对齐、发音偏差、节奏”

如果现在把 AI 输出设计为单一结构，后续会频繁破坏兼容。

### 2.3 把内容源只理解为词库
当前只有词语卡片，但未来内容源会扩展成：
- 词语库
- AI 动态生成词语
- 指定领域词语包
- 辩题库
- 演讲稿库
- TED 稿件库
- 自定义训练素材

如果内容层没有抽象，后面每加一个内容类型就要复制一套管理逻辑。

### 2.4 把前台和后台混在一起
未来你要管理：
- 注册用户数
- 活跃用户
- 训练完成率
- 付费情况
- 词库内容
- 辩题内容
- 演讲稿内容
- 内容难度分层

如果现在不预留后台能力，后面就会出现“业务系统已经上线，但没有运营抓手”的情况。

### 2.5 把音频只当输入，不当资产
V1 你明确说不存音频，这个方向没有问题。

但未来如果要支持：
- 语音分析
- 语音播报
- 跟读对比
- 演讲训练

那技术上必须把“音频处理链路”单独抽象出来。即使 V1 不存原始音频，也要在架构上把语音能力从普通文本输入里分离出来。

## 3. 长期正确架构目标

开口应该从一开始就围绕这 6 个核心域设计：

1. 用户与账号域
2. 训练引擎域
3. 内容素材域
4. AI 能力编排域
5. 支付与权限域
6. 数据分析与后台运营域

这 6 个域是长期稳定的，后续无论增加多少训练玩法，基本都还是在这几个域内扩展。

## 4. 推荐总体架构

我不建议你一开始就上微服务。现阶段最合适的是：

**模块化单体 + 清晰领域分层 + 可拆分接口边界**

也就是：
- 部署上先保持一个后端服务，控制复杂度
- 代码结构上按领域拆模块，不要按页面堆接口
- 数据模型上做统一抽象，避免只适配卡片玩法
- AI、语音、支付、后台这些外部能力都走独立适配层

这样做的好处：
- V1 开发成本低
- V2/V3 继续扩展时不需要推翻
- 将来用户量上来后，也容易按模块拆服务

## 5. 推荐架构分层

建议采用下面这套结构：

```text
微信小程序前台
管理后台 Web
        │
        ▼
API Gateway / BFF
        │
        ▼
应用服务层
        │
        ├── 用户与权限模块
        ├── 训练引擎模块
        ├── 内容素材模块
        ├── AI 编排模块
        ├── 语音能力模块
        ├── 支付订单模块
        └── 数据分析模块
        │
        ▼
基础设施层
        ├── MySQL / PostgreSQL
        ├── Redis
        ├── 对象存储（未来可选）
        ├── 消息队列（未来可选）
        ├── AI 模型服务
        ├── ASR/TTS 服务
        └── 微信开放能力
```

## 6. 关键设计原则

## 6.1 把“训练模式”抽象成一级概念

不要把系统理解成“卡片训练系统”，而是理解成：

`训练模式 + 内容素材 + 用户输入 + AI 反馈 + 训练记录`

建议预设的训练模式枚举：
- `card_association` 卡片联想
- `debate` AI 辩论
- `speech_reading` 演讲朗读
- `speech_imitation` 演讲模仿
- `free_expression` 自由表达

即使 V1 只有 `card_association`，表结构和接口也要允许未来扩展到多模式。

## 6.2 把“内容素材”抽象成统一内容中心

未来所有训练内容都不应该散落在不同表里，而应统一抽象为内容实体。

建议内容类型：
- `word`
- `debate_topic`
- `speech_script`
- `speech_audio`
- `quote`
- `exercise_pack`

建议内容公共字段：
- `content_id`
- `content_type`
- `title`
- `body`
- `content_source_type`
- `difficulty_level`
- `tags`
- `status`
- `source`
- `created_by`
- `created_at`

这样未来管理后台可以统一管理内容，而不是分别管理几十张碎表。

这里建议把内容来源显式分成 3 类：
- `preset` 预置内容
- `ai_generated` AI 生成内容
- `imported` 外部导入内容

这样未来“指定领域生成词语”时，不需要推翻内容中心，只是新增生成入口和来源标记。

## 6.3 把“训练会话”作为核心实体

长期来看，最核心的数据不应该只是“训练记录”，而应该是“训练会话”。

建议核心实体：

### `training_session`
- `session_id`
- `user_id`
- `mode`
- `status`
- `difficulty_level`
- `started_at`
- `ended_at`
- `duration_seconds`

### `training_session_content`
- `id`
- `session_id`
- `content_id`
- `content_type`
- `display_order`
- `snapshot_json`

说明：
- `snapshot_json` 用于记录当次训练所用素材快照
- 避免后续内容库修改后，历史记录无法还原

### `training_submission`
- `submission_id`
- `session_id`
- `input_type` (`text` / `speech` / `mixed`)
- `raw_text`
- `transcribed_text`
- `audio_meta_json`
- `submitted_at`

### `training_feedback`
- `feedback_id`
- `session_id`
- `feedback_schema`
- `feedback_json`
- `overall_score`
- `generated_at`

这样一来：
- 卡片训练可以挂 2 个词
- 辩论训练可以挂 1 个辩题
- 演讲训练可以挂 1 篇稿件 + 1 份参考音频

都不需要重做主干模型。

## 6.4 AI 输出必须走“反馈 schema”

不同训练模式的反馈结构不同，所以不能只有一套固定字段。

建议设计：
- 每种模式绑定一个 `feedback_schema`
- 前端按 schema 渲染不同反馈区块
- 通用字段统一保留：
  - `overall_score`
  - `summary`
  - `strengths`
  - `weaknesses`
  - `suggestions`

示例：

```json
{
  "feedback_schema": "card_association_v1",
  "overall_score": 7.8,
  "summary": "你已经建立了基本联系，但论证还不够具体。",
  "details": {
    "structure_clarity": 7,
    "logic_connection": 8,
    "specificity": 6,
    "fluency": 8
  }
}
```

以后辩论模式可以变成：

```json
{
  "feedback_schema": "debate_v1",
  "overall_score": 8.1,
  "summary": "你的立场明确，但反驳对方观点的力度还不够。",
  "details": {
    "position_clarity": 9,
    "argument_strength": 8,
    "rebuttal_quality": 6,
    "response_speed": 8
  }
}
```

这样不是推翻，而是扩展。

## 6.5 语音能力单独抽象

未来你要做：
- ASR 语音转文字
- TTS 语音播报
- 发音分析
- 音频对齐
- 跟读评测

所以语音相关能力不能散落在训练页里调用。

建议单独设计 `speech_service`，职责包括：
- 录音元信息接收
- ASR 调用
- TTS 调用
- 音频分析任务下发
- 语音能力结果标准化

V1 虽然不存原始音频，但建议先预留：
- `audio_meta_json`
- `speech_task_id`
- `tts_task_id`

未来扩展时就不会重新洗表。

## 6.6 后台运营从一开始就预留

后台不要等上线后再想。

至少从数据层先预留出这几类能力：

### 用户运营
- 总注册用户
- 日活 / 周活 / 月活
- 新用户次留
- 训练完成率

### 付费运营
- 付费人数
- 付费转化率
- 订单数
- 续费率
- 收入趋势

### 内容运营
- 词库数量
- AI 生成词数量
- 内容使用次数
- 不同难度使用分布
- 高分素材 / 低分素材

### 训练运营
- 各训练模式使用次数
- 平均时长
- 平均得分
- 各模式完成率

因此建议从 V1 就有一个 `analytics_event` 或事实表体系，哪怕先做简版。

不建议继续只围绕以下表扩展：
- `card`
- `round`

建议升级为下面这组主干模型：

## 7.1 用户域
- `user`
- `user_profile`
- `user_membership`
- `user_login_log`

## 7.2 内容域
- `content_item`
- `content_tag`
- `content_difficulty`
- `content_bundle`

## 7.3 训练域
- `training_session`
- `training_session_content`
- `training_submission`
- `training_feedback`
- `training_progress`

## 7.4 支付域
- `membership_order`
- `payment_transaction`
- `membership_rights_log`

## 7.5 运营分析域
- `analytics_event`
- `daily_user_stats`
- `daily_revenue_stats`
- `daily_training_stats`

## 7.6 后台管理域
- `admin_user`
- `admin_role`
- `operation_log`

## 8. 推荐接口设计方向

API 也不该继续只围绕“卡片接口”来长。

建议按领域分组：

### 用户与会员
- `/api/auth/*`
- `/api/user/*`
- `/api/membership/*`

### 内容中心
- `/api/content/*`

### 训练引擎
- `/api/training/session/create`
- `/api/training/session/detail`
- `/api/training/session/submit`
- `/api/training/session/feedback`

### 卡片训练子能力
- `/api/modes/card-association/*`

### 辩论训练子能力
- `/api/modes/debate/*`

### 演讲训练子能力
- `/api/modes/speech/*`

### 语音能力
- `/api/speech/asr`
- `/api/speech/tts`
- `/api/speech/analyze`

### 后台
- `/admin-api/users/*`
- `/admin-api/orders/*`
- `/admin-api/content/*`
- `/admin-api/analytics/*`

这样新增模块时，不会把现有接口体系打乱。

## 9. 推荐前端架构方向

未来会有两个前端：

1. 微信小程序前台
2. Web 管理后台

### 小程序前台
建议按“训练模式”拆页面和状态管理，而不是全部堆在一个训练页里。

建议结构：
- 首页
- 模式选择页
- 卡片训练页
- 辩论训练页
- 演讲训练页
- 历史记录页
- 会员页

### 管理后台
建议模块：
- 仪表盘
- 用户管理
- 订单管理
- 内容管理
- 训练分析
- 配置中心

## 10. 推荐 AI 架构方向

未来 AI 能力不会只有一个“分析文本”接口。

建议把 AI 能力分成 4 层：

### 10.1 Prompt 模板层
按训练模式管理：
- 卡片联想 prompt
- 辩论 prompt
- 演讲评分 prompt
- 跟读分析 prompt

### 10.2 任务编排层
负责决定：
- 调哪个模型
- 是否同步返回
- 是否重试
- 是否走异步任务

### 10.3 结果标准化层
把不同模型返回统一转成平台内部 schema

### 10.4 成本控制层
负责：
- 免费版 / 会员版模型差异
- 长文本截断
- 限频
- 重试策略

### 10.5 内容生成层
除了“分析用户表达”，AI 未来还要承担“生成训练素材”。

建议从现在就把这类能力纳入 AI 架构：
- 生成生活化词语
- 按指定领域生成词语
- 生成指定难度词语
- 去重后补齐词池

推荐生成参数：
- `content_type`
- `generation_scene`
- `difficulty_level`
- `domain_tag`
- `count`
- `dedup_scope`

这样未来 AI 生成词库不是旁路能力，而是平台内容中心的一部分。

## 11. 难度体系应该怎么接

你提到后续要支持词语难度选择，这说明难度不是卡片独有能力，而应是平台级能力。

建议难度体系统一设计：
- `beginner`
- `intermediate`
- `advanced`

每种内容都可挂难度：
- 词语有难度
- 辩题有难度
- 演讲稿有难度

训练会话也有难度：
- `training_session.difficulty_level`

这样后续所有玩法都能共用“难度选择”。

## 12. 语音播报与语音分析怎么接

你未来要“能直接分析语音，能播报语音”，这意味着要从现在区分三种数据：

1. 用户输入内容
2. 语音处理结果
3. 播报输出内容

建议拆成：

### 用户输入
- 原始文本
- ASR 文本
- 输入时长

### 分析结果
- 发音准确率
- 停顿分布
- 语速
- 情绪或感染力标签

### 播报结果
- TTS 音频任务
- 播报音色
- 播报文本版本

这样未来做演讲跟读时就能直接复用。

## 12.5 AI 生成词库是否需要现在改

我的建议是：

### 现在就要改的
- 内容模型里增加“来源类型”概念
- 后台里预留“内容生成”入口
- AI 层预留“内容生成任务”能力
- 训练引擎允许从“预置内容池”或“动态生成结果”取词

### 现在不用做得很重的
- 不需要一开始就做完整自动词库生产系统
- 不需要一开始就做复杂审核流
- 不需要一开始就让所有训练模式都支持动态生成

一句话：

**能力边界现在要抽象，功能复杂度可以后面再加。**

## 13. 是否需要微服务

现在不需要。

我建议分三阶段：

### 阶段一：模块化单体
适合 V1 到 V2 初期

### 阶段二：拆出高耦合外部能力
优先可拆：
- AI 编排服务
- 语音服务
- 后台分析任务

### 阶段三：按业务域拆服务
当用户量、内容量、任务量明显上来后，再考虑拆：
- 用户服务
- 训练服务
- 内容服务
- 支付服务
- 分析服务

## 14. 当前最容易做错的点

如果现在开发时出现以下做法，后续就容易返工：

### 不建议
- 训练记录表只为卡片模式设计
- 所有 AI 结果共用一套死字段
- 所有训练内容都分别建散表
- 语音逻辑直接写在页面调用里
- 后台数据靠手工 SQL 查
- 接口只按页面命名，不按领域命名

### 建议
- 先统一训练会话模型
- 先统一内容模型
- 先统一反馈 schema
- 先统一语音能力入口
- 先预留后台统计事件

## 15. 一套能支持长期版本的推荐方案

如果你希望现在就选一个能支持长期版本迭代的架构，我给你的建议是：

### 核心形态
**微信小程序 + Web 管理后台 + 模块化单体后端**

### 核心数据主线
**用户 -> 内容 -> 训练会话 -> 用户提交 -> AI 反馈 -> 分析统计**

### 核心平台抽象
- 训练模式
- 内容中心
- AI 反馈 schema
- 语音能力中心
- 会员与权限中心
- 运营分析中心

这套抽象足以支撑：
- 当前的卡片联想训练
- 后续的难度分层
- 语音分析
- 语音播报
- AI 辩论
- 演讲朗读
- TED 稿件引入
- 管理后台

而且不需要现在就上过重架构。

## 16. 对你当前方案的最终判断

我的判断是：

### 不需要推翻的部分
- 产品方向
- 微信小程序形态
- 首发玩法选择
- 会员体系
- 训练主流程

### 需要升级抽象的部分
- 数据模型
- API 设计
- AI 反馈结构
- 内容管理方式
- 语音能力接入方式
- 后台统计与运营能力

一句话说：

**你的 V1 方案可以保留，但技术实现必须从“单玩法项目”升级成“多模式表达训练平台”。**
