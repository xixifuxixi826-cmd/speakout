# 开口技术选型与开发里程碑

## 1. 文档目标

这份文档回答 4 个问题：

1. 这个项目用什么技术栈最合适
2. 前后端代码目录应该怎么搭
3. V1 到长期版本该怎么分阶段开发
4. 现在开始做，最稳的落地顺序是什么

这份方案以你的目标为前提：
- 先做微信小程序首发
- 后续接管理后台
- 长期扩展多训练模式
- 保证架构可持续迭代

## 2. 总体技术建议

我推荐的整体方案是：

**微信小程序前台 + Web 管理后台 + Java Spring Boot 模块化后端 + MySQL + Redis**

如果你更偏快速试错，也可以选 Node.js，但从你后续会接：
- 支付
- 后台
- 多模式训练
- AI 编排
- 语音任务
- 数据统计

这些需求看，长期稳定性和团队协作上，Java 会更适合做主后端。

## 3. 技术选型建议

## 3.1 小程序前台

推荐：
- 微信原生小程序 或 `Taro`

建议判断：
- 如果你只做微信小程序：优先微信原生
- 如果未来还可能扩展 H5 / 多端：优先 `Taro`

当前阶段我更建议：
- **V1 先用微信原生小程序**

原因：
- 接微信登录、支付、录音能力最直接
- 学习和调试链路最短
- 首发成本更低

推荐技术：
- 小程序原生框架
- TypeScript
- Zustand 风格的轻状态管理思路，或按页面局部状态为主

## 3.2 管理后台

推荐：
- `React + Ant Design`

原因：
- 后台型产品成熟稳定
- 表格、筛选、图表、权限页比较顺手
- 开发效率高

推荐组合：
- React
- TypeScript
- Ant Design Pro 或 Ant Design 基础版
- React Router
- TanStack Query

## 3.3 后端

推荐：
- `Java 21 + Spring Boot`

推荐组合：
- Spring Boot
- Spring Web
- Spring Validation
- Spring Security
- MyBatis-Plus 或 JPA
- Redis
- MySQL 8

我更建议：
- **Spring Boot + MyBatis-Plus**

原因：
- 后台管理、业务表、查询表很多
- 未来统计类接口较多
- 对中国团队协作和维护更友好

## 3.4 数据库

推荐：
- `MySQL 8`

原因：
- 成熟稳定
- 后台查询、运营分析、订单系统都适配
- 团队和云服务支持广

## 3.5 缓存与会话

推荐：
- `Redis`

用途：
- 登录态缓存
- 用户每日额度缓存
- 高频配置缓存
- 训练过程中的短期状态缓存

## 3.6 对象存储

V1 可选，不一定马上上。

推荐未来预留：
- 腾讯云 COS 或阿里云 OSS

用途：
- 演讲稿附件
- 参考音频
- TTS 输出音频
- 未来语音分析相关文件

## 3.7 AI 与语音接入

建议通过“适配层”统一接入，不把厂商 SDK 直接散在业务代码里。

推荐抽象：
- `AiProvider`
- `SpeechProvider`

AI 可能接：
- OpenAI 兼容接口
- 国内大模型接口

语音可能接：
- 微信生态能力
- 腾讯云 ASR / TTS
- 其他语音服务商

关键原则：
- 业务层只认平台接口，不直接依赖某一个厂商

## 4. 为什么推荐这套技术栈

## 4.1 不推荐一开始就微服务

原因：
- 业务还在变化期
- 团队成本会被基础设施吃掉
- 调试链路太长

更适合的是：
- 单体部署
- 模块分层
- 明确领域边界

## 4.2 不推荐一开始上很重的前端架构

原因：
- 现在核心难点在业务抽象，不在炫技型前端
- 小程序端优先保证录音、流程和稳定性

## 4.3 不建议把 AI 当成一个直接调用接口

应该有：
- Prompt 配置层
- 模型调用层
- 结果标准化层
- 失败重试层

否则后续换模型、做会员分层、做不同训练模式会很痛。

## 5. 推荐项目结构

## 5.1 工作区建议

建议项目最终拆成 3 个仓库或 1 个 monorepo 的 3 个应用：

```text
kaikou/
├── miniapp/
├── admin-web/
└── server/
```

如果团队不大，我建议：
- 可以一个大目录管理
- 但代码逻辑仍按 3 个应用拆分

## 5.2 小程序目录建议

```text
miniapp/
├── src/
│   ├── pages/
│   │   ├── home/
│   │   ├── training-card/
│   │   ├── training-speaking/
│   │   ├── feedback/
│   │   ├── history/
│   │   └── membership/
│   ├── components/
│   ├── services/
│   ├── stores/
│   ├── utils/
│   ├── constants/
│   └── types/
├── project.config.json
└── package.json
```

说明：
- `pages` 放页面
- `components` 放复用 UI 组件
- `services` 放接口调用
- `stores` 放状态管理
- `types` 放接口和数据结构类型

## 5.3 管理后台目录建议

## 7. 环境规划建议

至少分 3 套环境：

1. 本地开发环境
2. 测试环境
3. 生产环境

## 7.1 开发环境
- 本地 MySQL
- 本地 Redis
- 小程序开发者工具
- 使用测试 AI Key 和测试支付配置

## 7.2 测试环境
- 云数据库
- 云 Redis
- 测试域名
- 后台联调环境

## 7.3 生产环境
- 正式微信小程序配置
- 正式支付配置
- 正式 AI / 语音配置
- 日志和监控体系

## 8. 开发阶段建议

## 第一阶段：V1 核心闭环

目标：
- 跑通“登录 -> 训练 -> AI 反馈 -> 会员 -> 历史”的主链路

范围：
- 微信登录
- 首页
- 卡片训练
- 文本提交
- 语音转文字
- AI 反馈
- 历史记录
- 会员支付

不追求：
- 完整后台
- 复杂统计
- 多训练模式

## 第二阶段：运营和稳定性补强

目标：
- 让产品能看数、能管内容、能调策略

范围：
- 管理后台基础版
- 内容管理
- 用户管理
- 订单管理
- 仪表盘
- Prompt 配置
- 日志与告警

## 第三阶段：平台化扩展

目标：
- 开始真正进入多训练模式平台

范围：
- 难度选择
- AI 辩论
- 演讲训练
- TTS 播报
- 语音分析
- 内容包系统

## 9. 开发里程碑建议

下面这个节奏比较稳。

## M1：需求和架构冻结

产出：
- PRD 确认
- 平台架构确认
- 数据库设计确认
- API 设计确认

完成标准：
- 团队对主干模型无分歧

## M2：基础工程搭建

产出：
- 小程序项目初始化
- 后端项目初始化
- 管理后台项目初始化
- CI/CD 初版
- 开发环境可运行

完成标准：
- 三端都能本地启动

## M3：账号与会员主链路

产出：
- 微信登录
- 用户信息接口
- 会员状态接口
- 支付下单与回调

完成标准：
- 用户可登录并识别会员身份

## M4：卡片训练闭环

产出：
- 卡片轮次初始化
- 翻卡
- 选卡
- 提交训练
- 历史记录写入

完成标准：
- 用户可完成一次完整训练

## M5：AI 与语音接入

产出：
- ASR 接入
- AI 分析接入
- 固定反馈结构
- 重试与异常处理

完成标准：
- 用户可通过语音或文字完成分析

## M6：后台最小运营版

产出：
- 后台登录
- 仪表盘概览
- 用户列表
- 订单列表
- 词语管理

完成标准：
- 运营可查看核心数据并维护内容

## M7：首发准备

产出：
- 联调完成
- 压测与异常修复
- 小程序提审
- 生产部署

完成标准：
- 可上线

## 10. 推荐排期思路

如果是一个小团队，建议顺序是：

1. 后端先把用户、训练、会员主表和 API 骨架搭好
2. 小程序同步开发首页和训练主流程
3. 先用文本提交流程打通 AI 反馈
4. 再接 ASR
5. 最后补历史记录和会员支付
6. 后台放在 V1 中后段接入

原因：
- 这样最先验证的是核心价值，不是外围能力
- ASR 和支付通常更容易拖进度
- 文本链路先跑通，能快速验证训练产品是否成立

## 11. 团队角色建议

如果有条件，最舒服的配置是：

1. 产品 / 项目负责人
2. 小程序前端 1 人
3. 后端 1 人
4. 后台前端 1 人
5. UI 设计 1 人

如果团队更小，也可以：

1. 产品兼项目
2. 前端 1 人，兼小程序和后台
3. 后端 1 人

## 12. 当前最值得坚持的原则

从现在开始，整个项目最重要的是守住这 5 件事：

1. 不按单玩法写死数据结构
2. 不按页面堆接口
3. 不把 AI 厂商直接写进业务代码
4. 不把语音能力散落到页面层
5. 不等上线后再补运营后台

## 13. 最终建议

如果你现在就准备进入开发，我建议直接按下面这条路线走：

### 技术栈
- 小程序：微信原生 + TypeScript
- 后台：React + Ant Design
- 后端：Spring Boot + MyBatis-Plus
- 数据库：MySQL
- 缓存：Redis

### 架构形态
- 模块化单体
- 平台级数据模型
- 模式化训练引擎

### 开发顺序
- 先做卡片训练闭环
- 再做会员支付和历史
- 再做后台
- 再做辩论、演讲、语音分析等扩展能力

这条路线能兼顾：
- 首发速度
- 长期扩展
- 架构稳定性
